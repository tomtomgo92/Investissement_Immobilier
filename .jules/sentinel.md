## 2026-02-25 - Validate Shared Simulation Data
**Vulnerability:** Missing Input Validation in Shared URL Decoding. The application parsed arbitrary JSON from URL fragments (`window.location.hash`) and hydrated state directly without structural validation, leading to potential DoS or logic errors.
**Learning:** Client-side hydration from URLs is an often-overlooked attack surface. `JSON.parse` is not sufficient validation. Even without direct execution, malformed data can cause crashes or unexpected application states.
**Prevention:** Implement strict schema validation (checking types and required fields) for all external inputs before using them in application state.
## 2024-05-24 - [DoS via Missing Deep Validation in Shared Arrays]
**Vulnerability:** A malicious actor could craft a share URL containing extremely long string values within deeply nested array items (e.g., `sim.data.charges[0].name`). Since validation only checked the `sim.name` length and array item count (max 100), passing 100 items with 1MB string names could lead to memory exhaustion and browser crashes on the victim's machine during deep copy operations or state rendering.
**Learning:** Checking top-level properties and array limits is insufficient if the internal structure of the array elements can hold unbounded data. Shallow validation gives a false sense of security for deep objects. In this codebase, shared state is parsed and then directly injected into complex React state, making it highly susceptible to DoS.
**Prevention:** Apply strict type and length boundary checks (e.g., `typeof c.name === 'string' && c.name.length <= MAX_NAME_LENGTH`) to *every* relevant nested string property when validating external JSON payloads before they reach application state. Use safe fallbacks (e.g., allow missing properties if acceptable `!c.name || ...`) to avoid breaking backward compatibility with valid legacy data.
